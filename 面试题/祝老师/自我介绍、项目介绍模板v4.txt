    1. 服务器的性能调优：
        有一次，我们刚升级上线一个功能，就有用户反应速度变慢了，我们赶紧查日志，发现是______接口服务变慢了，这个接口服务不是这次升级的，怎么会影响到这里呢，我们继续查下去，发现这个接口用到的一个缓存里的数据，被新功能删掉了，但是没有及时更新回去，导致出现了问题，我们测试那个新功能时，没有测到这一点，导致这个接口变慢了，发现是这个问题后，我们立即回滚了新功能代码，重新在线下认真测试，确保没有问题后，重新上线，解决了这个问题。
    2. 用 NginX 实现负载均衡：
        我们一开始，只使用了一个python服务器，随着用户的增长，这个服务器越来越慢，我们就要想办法加服务器，当时我们团队里都没有什么经验，我主动申请解决这个问题，当时查了很多资料，又向其他团队有经验的同事认真请教，最终我们确定了使用Nginx做负载均衡的方案，接着我仔细去学习了Nginx如何做负载均衡，使用upstream方式，可以让 Nginx 可以向后边的多台服务器转发请求，我们增加了第二台 python服务器，在Nginx 上用权重方式向后分发，解决了负载均衡的问题。
    3. 数据库 Sharding：
        我们业务的用户量大概2000万，导致用户访问越来越慢，为了解决用户访问慢的问题，需要对数据库做 Sharding，我们在网上查找了数据库分拆的方案，又向资深研发做了请教，确定hash(username)方式分拆，把user表分拆为256个，每个表了存的数据只有七八多万，虽然表变多了，但每个表只有几万数据，性能得到了极大提升

    5. 解决了一个诡异的bug：
        有一次，我们碰到一个诡异的问题，每到晚上11点半，_____接口就不能正常工作，查了很久，也找不到问题，从日志，从数据库，从程序都无法发现问题，后来才发现，是这个接口写了一个日志文件，这个日志文件挂在了一个网络磁盘上，每到11点半，网络里有一个很占带宽的备份服务就开始工作，占满了带宽，导致这个日志迟迟写不成功，进而影响了接口功能，发现问题后，我们改变了写日志的策略，先把日志写入本地，再同步到中心服务器，这样就避免了这个问题。

    7. 采用数据库代理方案，提升数据库性能：
        我们业务一开始实现的时候，数据库就一台单机，也没考虑未来性能不够该怎么办，结果，终于数据库越来越慢，有一天领导忍不了了，让我解决问题，我查了很多方案，最终选定用数据库代理方式解决问题，这种方式可以实现主从方案，同时又不需要程序改代码，正好符合我们的需求，于是我们对比之后，选择了京东开源的一款数据库代理产品，这款产品已经成为 apache 的开源项目，发展势头良好，开发也比较活跃，持续在更新功能，比较适合我们业务要求，于是我们按照京东的项目，开始动手搭建数据库代理环境，按照配置文档，配好代理，连好主从数据库，python程序只需要修改一下数据库访问地址，就可以顺利用上主从分离的方案了，顺利的解决了数据库性能提升的问题。
    8. 用 Redis 实现排行榜：
        我们有一个业务，需要实现一个好友排行榜，这个排行榜不是全服一个的，是每人一个榜单，我们寻找了一下实现方案，发现用 Redis 实现排行榜简单高效，于是我们也动手开始用 redis 实现我们的排行榜，写入榜单时，每个人一个榜单，每个榜单都是zset，一旦一个人的分数有了变化，就会插入他每个好友的榜单中，这样当一个人要查看好友榜单时，只需要倒序从zset中找出最高的几个人的id，再拼装好用户数据返回去就可以了。
    9. 用 Redis 做限流、抢号
        在一个业务场景，我们有一个排队抢号的业务，只有300个号，先来先得，为了实现这个抢号业务，我们在redis里保存一个值300，每被取走一个号，就decr这个值，由于 decr 指令操作是原子的，所以确保不会给两个人发到一样的号，每有一个人取走一个号，这个值就减1，直到这个值变为0，所有300个号抢光，不再发出新号。拿到号的进程才能继续进行后边的业务流程，由此实现了一个限流的业务。
    10. 用 Redis 做发号器
        我们系统里有很多地方需要用到一个唯一的ID值，如果依赖数据库自增字段，可能会产生id冲突，为了解决此问题，我们用 Redis 实现了一个发号器，设置一个 String 字段，初始值为0，每有一个进程来申请号码，就 incr 这个字段，返回一个新值出去，由于 incr 操作是原子的，确保不会有两个进程拿到一样的号，这样一直不断一直发新号，前边再加上业务识别标记，日期时间，确保发出的每个号都是唯一不重复的。
    11. 用 日志 实现统计
        我们有个业务场景，要统计用户浏览商品页的的次数，希望由此来计算每个商品的页面转化率，一开始我们想直接把用户浏览商品页的数据直接入库，结果发现一下子就把数据库拖慢了，我们只能转而寻找其他解决方法，经过与资深开发同事交流，他建议我们改为记录日志，再从日志分析汇总数据入库，我们按照这个思路重新设计了统计方案，先把浏览数据记录到 Nginx 日志中，然后再用一个脚本去统计日志里的数据，再把攻击结果汇总写入数据库，顺利的完成了这个任务。
